---
title: 第一方模式
description: 通过您的域名路由第三方脚本流量，以提升隐私和可靠性。
---

## 背景

当第三方脚本直接从外部服务器加载时，会暴露您的用户数据：

- **IP 地址暴露** — 每个请求都会向第三方泄露用户的 IP 地址
- **第三方 Cookie** — 外部脚本可设置跨站追踪的 Cookie
- **广告拦截器干扰** — 隐私工具会阻止已知追踪域的请求
- **连接开销** — 额外的 DNS 查询和 TLS 握手会降低页面加载速度

### 第一方模式如何帮助

第一方模式通过您的域名路由所有脚本流量：

- **用户 IP 匿名化** — 在转发前，IP 地址被匿名到子网级别，第三方无法识别单个用户
- **减少设备指纹识别** — 屏幕分辨率、User-Agent 和硬件信息被泛化到常见类别
- **无第三方 Cookie** — 请求为同源，消除跨站追踪
- **兼容广告拦截器** — 请求看起来像第一方请求
- **加载更快** — 无需针对外部域的额外 DNS 查询

## 工作原理

启用第一方模式时：

1. **构建时**：脚本被下载且 URL 重写为本地路径（例如：`https://www.google-analytics.com/g/collect` → `/_scripts/c/ga/g/collect`）
2. **运行时**：Nitro 路由规则将来自本地路径的请求代理回原始端点

```
用户浏览器 → 您的服务器 (/_scripts/c/ga/...) → Google Analytics
```

您的用户永远不会直接连接第三方服务器。

## 使用方法

### 全局启用

为所有支持的脚本启用第一方模式：

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scripts: {
    firstParty: true,
    registry: {
      googleAnalytics: { id: 'G-XXXXXX' },
      metaPixel: { id: '123456' },
    }
  }
})
```

### 隐私模式

第一方模式通过 `privacy` 选项支持两个隐私等级：

| 模式 | 说明 |
|------|-------|
| `'anonymize'`（默认） | IP 地址被匿名到子网级别，屏幕分辨率和硬件信息泛化为常见类别，User-Agent 归一为浏览器类别 + 主版本。Analytics ID 保留，追踪功能仍有效。 |
| `'proxy'` | 请求按原样通过您的服务器转发。移除敏感请求头（cookies、授权），但不修改分析数据包。隐私来自第三方只看到您的服务器 IP，而非用户 IP。 |

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scripts: {
    firstParty: {
      privacy: 'proxy', // 或 'anonymize'（默认）
    }
  }
})
```

::callout{type="info"}
在 `anonymize` 模式中，指纹数据是**泛化**而非剥离 —— 分析端点仍收到有效数据，只是精度降低。例如，屏幕分辨率 `1440x900` 会变成 `1920x1080`（桌面类别），User-Agent 归一为 `Mozilla/5.0 (compatible; Chrome/131.0)`。
::

### 自定义路径

自定义代理端点路径：

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scripts: {
    firstParty: {
      collectPrefix: '/_analytics', // 默认：/_scripts/c
    }
  }
})
```

### 按脚本选择退出

为某个特定脚本禁用第一方路由：

```ts
useScriptGoogleAnalytics({
  id: 'G-XXXXXX',
  scriptOptions: {
    firstParty: false, // 直接从 Google 加载
  }
})
```

## 要求

第一方模式需要**服务器运行时环境**。它无法用于完全静态托管（例如 `nuxt generate` 部署到 GitHub Pages），因为代理端点需要服务器转发请求。

对于静态部署，仍可启用第一方模式 —— 脚本会包含重写的 URL，但您需要手动配置托管平台的重写规则。

### 静态托管重写规则

如果静态部署，请配置平台代理这些路径：

```
/_scripts/c/ga/* → https://www.google.com/*
/_scripts/c/gtm/* → https://www.googletagmanager.com/*
/_scripts/c/meta/* → https://connect.facebook.net/*
```

## 第一方模式 vs bundle 选项

第一方模式替代 `bundle` 选项：

| 功能 | `bundle: true` | `firstParty: true` |
|-------|-------------|-------------------|
| 构建时下载脚本 | ✅ | ✅ |
| 从您域名提供脚本 | ✅ | ✅ |
| 重写收集 URL | ❌ | ✅ |
| 代理 API 请求 | ❌ | ✅ |
| 隐藏用户 IP | ❌ | ✅ |
| 阻止第三方 Cookie | ❌ | ✅ |

`bundle` 选项只自托管脚本文件，第一方模式还会重写并代理所有收集/追踪端点，实现完整的第一方路由。

::callout{type="warning"}
`bundle` 选项已废弃。新项目请使用 `firstParty: true`。
::

## 默认行为

第一方模式**默认启用**。对于静态托管（如 GitHub Pages），脚本仍打包，但您需配置平台重写以使代理端点生效。

关闭第一方模式：

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scripts: {
    firstParty: false
  }
})
```

### Cloudflare Pages

在公共目录创建一个 `_redirects` 文件：

```txt [public/_redirects]
/_scripts/c/ga/* https://www.google.com/:splat 200
/_scripts/c/ga-legacy/* https://www.google-analytics.com/:splat 200
/_scripts/c/gtm/* https://www.googletagmanager.com/:splat 200
/_scripts/c/meta/* https://connect.facebook.net/:splat 200
```

或者使用 Cloudflare Workers 以获得更多控制：

```ts [functions/_scripts/c/[[path]].ts]
export const onRequest: PagesFunction = async (context) => {
  const url = new URL(context.request.url)
  const path = url.pathname.replace('/_scripts/c/', '')

  // 根据前缀路由
  const routes: Record<string, string> = {
    'ga/': 'https://www.google.com/',
    'gtm/': 'https://www.googletagmanager.com/',
    'meta/': 'https://connect.facebook.net/',
  }

  for (const [prefix, target] of Object.entries(routes)) {
    if (path.startsWith(prefix)) {
      const targetUrl = target + path.slice(prefix.length) + url.search
      return fetch(targetUrl, context.request)
    }
  }

  return new Response('Not found', { status: 404 })
}
```

## 故障排查

### 分析无数据追踪

**症状**：统计分析面板没有事件数据。

**检查项**：

1. **验证代理路由是否生效** — 开发模式下检查 `/_scripts/status.json` 或 DevTools 的 Scripts 面板
2. **查看浏览器网络面板** — 观察是否有对 `/_scripts/c/` 路径的请求且返回 200
3. **检查服务器日志** — 查找 Nitro 代理错误
4. **验证路由规则** — 运行 `npx nuxi build` 并检查 `.output/server/chunks/routes/rules.mjs`

```ts
// 调试：在服务器中间件中打印所有请求
export default defineEventHandler((event) => {
  if (event.path.startsWith('/_scripts/c/')) {
    console.log('代理请求:', event.path)
  }
})
```

### 跨域请求错误（CORS）

**症状**：浏览器控制台显示分析请求的 CORS 错误。

**解决方式**：

1. 确保 `nuxt.config.ts` 中正确配置了路由规则
2. 静态托管需确认平台重写设置正确
3. 检查重写目标 URL 完全匹配

### 缓存旧脚本

**症状**：脚本内容过时或 URL 未重写。

**解决方式**：

1. 清除构建缓存：

   ```bash
   rm -rf .nuxt/cache/scripts
   npx nuxi build
   ```

2. 开发阶段强制重新下载：

   ```ts
   useScriptGoogleAnalytics({
     id: 'G-XXX',
     scriptOptions: { bundle: 'force' }
   })
   ```

### 静态构建不代理

**症状**：脚本加载正常，但静态托管的分析无数据。

**这是预期行为** — 静态构建无法代理请求。请配置平台重写（见上方“平台重写”章节），或改用服务端渲染模式。

### 脚本下载失败

**症状**：构建时报网络超时错误。

**解决方案**：

1. 启用回退模式：

   ```ts [nuxt.config.ts]
   export default defineNuxtConfig({
     scripts: {
       firstParty: true,
       assets: {
         fallbackOnSrcOnBundleFail: true
       }
     }
   })
   ```

2. 增加超时时间：

   ```ts [nuxt.config.ts]
   export default defineNuxtConfig({
     scripts: {
       assets: {
         fetchOptions: {
           timeout: 30000 // 30 秒
         }
       }
     }
   })
   ```

## 常见问题

### 第一方模式会绕过 GDPR 同意要求吗？

**不会。** 第一方模式改变请求的路由方式，但不影响是否进行追踪。您仍需在加载追踪脚本前获得用户同意。使用同意触发器：

```ts
const { accept } = useScriptTriggerConsent()

// 仅在用户同意后加载
function onConsentGiven() {
  accept()
}
```

### 第三方修改了脚本，分析还会继续工作吗？

**会，且支持自动更新。** 脚本默认缓存 7 天，缓存过期后会重新下载并重写 URL。您可以自定义缓存时长：

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scripts: {
    assets: {
      cacheMaxAge: 86400000 // 1 天，单位毫秒
    }
  }
})
```

### 可以自定义代理路径吗？

**可以。** 使用 `collectPrefix` 选项：

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  scripts: {
    firstParty: {
      collectPrefix: '/_tracking' // 替代默认的 /_scripts/c
    }
  }
})
```

### 这支持服务器端追踪吗？

第一方模式适用于客户端脚本。服务器端追踪（如 Measurement Protocol）请直接从服务器端发送请求，无需代理层。

### 如何调试被代理的内容？

1. **DevTools**：打开 Nuxt DevTools → Scripts → 第一方标签页
2. **状态端点**：开发时访问 `/_scripts/status.json`
3. **控制台日志**：启用调试模式：

   ```ts [nuxt.config.ts]
   export default defineNuxtConfig({
     scripts: {
       debug: true
     }
   })
   ```

### 会影响性能吗？

**影响极小。** 代理带来少量延迟（约 10-50ms），但通常因以下因素补偿：

- 消除第三方域的 DNS 查询
- 绕过可能阻止请求的广告拦截器
- 降低连接开销（重用连接）

### 哪些脚本支持第一方？

当前支持文档「支持的脚本」中列出的 10 个脚本。对其他脚本，您可以：

1. 通过 Issue 请求支持
2. 使用（已废弃的）`bundle` 选项自托管脚本
3. 手动配置自定义路由规则

## 混合渲染

第一方模式兼容 Nuxt 的混合渲染功能：

### 路由级 SSR

使用 `routeRules` 禁用特定路由 SSR 时，第一方模式依然有效，因为：

1. 脚本是在构建时打包的，不是运行时
2. 代理路由规则是 Nitro 全局配置
3. 收集请求仍通过您的服务器

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  routeRules: {
    '/dashboard/**': { ssr: false }, // 仪表盘 SPA 模式
  },
  scripts: {
    firstParty: true, // 所有路由仍生效
  }
})
```

### ISR（增量静态再生成）

ISR 页面同样适用第一方模式。打包脚本作为静态资源提供，收集请求通过 Nitro 在运行时代理。

### Edge 渲染

第一方模式支持边缘部署（Cloudflare Workers、Vercel Edge 等）。代理路由使用 Nitro 原生代理，适用于各种部署目标。

::callout{type="info"}
对于边缘部署，请确保您的边缘运行时支持向分析端点发起外部 fetch 请求。
::

## 同意集成

第一方模式与同意管理互为补充：

- **第一方模式** 控制请求*发送的位置*（通过您的服务器而非直连）
- **同意触发器** 控制脚本*加载的时机*（同意之前不加载）

### 一起使用示例

```ts
<script setup>
// 第一方模式在 nuxt.config 全局配置
// 同意按脚本处理

const { status, accept } = useScriptTriggerConsent()

const { gtag } = useScriptGoogleAnalytics({
  id: 'G-XXXXXX',
  scriptOptions: {
    trigger: status, // 未同意前不加载
  }
})

function onConsentGiven() {
  accept() // 通过第一方代理加载脚本
}
</script>
```

### 同意横幅示例

```vue
<template>
  <div v-if="!hasConsent" class="cookie-banner">
    <p>我们使用分析数据来提升您的体验。</p>
    <button @click="acceptAll">接受</button>
    <button @click="rejectAll">拒绝</button>
  </div>
</template>

<script setup>
const hasConsent = ref(false)

const { accept: acceptGA } = useScriptTriggerConsent()
const { accept: acceptMeta } = useScriptTriggerConsent()

// 用同意触发器配置脚本
useScriptGoogleAnalytics({
  id: 'G-XXXXXX',
  scriptOptions: { trigger: useScriptTriggerConsent().status }
})

useScriptMetaPixel({
  id: '123456',
  scriptOptions: { trigger: useScriptTriggerConsent().status }
})

function acceptAll() {
  hasConsent.value = true
  acceptGA()
  acceptMeta()
  // 现在脚本通过第一方代理加载
}

function rejectAll() {
  hasConsent.value = true
  // 脚本不加载
}
</script>
```

### 隐私流

```
用户访问网站
       │
       ↓
脚本已注册但未加载（等待同意）
       │
       ↓
用户同意 Cookie
       │
       ↓
脚本从您的服务器加载（/_scripts/...）
       │
       ↓
分析请求通过您的服务器发送（/_scripts/c/...）
       │
       ↓
第三方看到的是您的服务器 IP，不是用户 IP
```

这既保证 GDPR 合规（先同意再追踪），也提升隐私保护（同意后第一方路由）。

## 健康检查

验证第一方模式是否正常：

### 1. 检查 DevTools

打开 Nuxt DevTools → Scripts → 第一方标签页，查看：

- 启用状态
- 配置的脚本
- 活跃的代理路由

### 2. 查看状态端点

开发时访问 `/_scripts/status.json`：

```json
{
  "enabled": true,
  "scripts": ["googleAnalytics", "metaPixel"],
  "routes": {
    "/_scripts/c/ga/**": "https://www.google.com/**",
    "/_scripts/c/meta/**": "https://connect.facebook.net/**"
  },
  "collectPrefix": "/_scripts/c"
}
```

### 3. 浏览器中验证

1. 打开浏览器 DevTools → 网络面板
2. 过滤路径 `/_scripts`
3. 触发分析事件
4. 确认请求目标为 `/_scripts/c/...`（非第三方域）
5. 检查响应状态为 200

### 4. CLI 状态

运行命令检查缓存状态：

```bash
npx nuxt-scripts status
```

显示已缓存脚本及大小。
